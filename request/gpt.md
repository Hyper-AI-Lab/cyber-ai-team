# Cyber-Team Reference Architecture for an AI-Run Digital Startup

## Executive conclusion

A digital-AI company operating system is technically achievable for a startup whose core work is already digital, but the correct design is **not** a single super-agent. The highest-confidence pattern in the current open-source ecosystem is a **compositional stack**: one layer for agent reasoning, one for durable workflow execution, one for memory, one for canonical business records, one for communications, and one for security and observability. The tools that exist today are already strong enough to cover most of that stack: LangGraph for stateful agent control and human interruption, Temporal for durable long-running workflows, MCP for agent-to-tool access, A2A for agent-to-agent interoperability, ERP/CRM/document systems for company records, and open telephony/messaging systems for external communications. citeturn25view0turn28view2turn39view1turn25view3turn25view9turn27view0turn25view5turn34view1turn25view6

The best practical architecture for your “cyber-team” is therefore a **company control plane** built around LangGraph plus Temporal; a **role factory** that creates agents from manifests rather than hard-coded prompts; a **layered memory system** using pinned memory, checkpoints, retrieval memory, and a real system of record; an **owner console** built around Open WebUI plus workflow/tracing dashboards; and a **governance plane** using Keycloak, OpenFGA, OPA, and a secrets manager. If you stay strict about license quality, prefer fully open-source components where possible, and treat open-core or fair-code tools as optional edge additions rather than architectural foundations. citeturn25view7turn30view0turn29view3turn29view1turn40search0turn29view2turn31view8turn35search0turn35search1turn35search2

## What the current open-source landscape already solves

For the core multi-agent layer, the strongest current choices are LangGraph, AG2, CrewAI, and Haystack, but they are good at different things. LangGraph is the strongest fit for a company OS because it gives low-level control over single-agent, multi-agent, and hierarchical flows, and pairs that with persistence, checkpoints, interrupts, and resumability. AG2 is strong for documented multi-agent chat patterns and shared-context collaboration. CrewAI is lean and role-oriented, which makes it good for rapid prototyping of specialist crews. Haystack is especially strong when retrieval, tools, and production pipeline structure are central. For a real company runtime, LangGraph is the best primary brain; AG2 and CrewAI are better treated as experimental or domain-specific satellites rather than the main control surface. citeturn25view0turn28view2turn39view1turn31view0turn31view1turn35search7turn31view3

For durable execution, agent frameworks alone are not enough. Temporal is built specifically for reliable, crash-tolerant long-running application workflows; it supports schedules, signals, updates, queries, and resumable message-driven workflow control. Prefect is a good open-source orchestration engine for Python-native data and automation flows. For visual automation, Activepieces is the cleaner open-source choice because its core is MIT-licensed, whereas n8n is explicitly fair-code under its Sustainable Use License rather than pure open source. That licensing difference matters if you want the core of the company OS to remain cleanly self-hostable and redistributable. citeturn25view3turn28view3turn28view4turn24search1turn30view9turn30view8turn35search2turn35search4turn35search0

At the interoperability layer, MCP and A2A are the decisive standards. MCP standardizes how agents connect to tools, resources, files, databases, and workflows. A2A standardizes how agents communicate with other agents. A2A’s current official documentation explicitly positions it as complementary to MCP: MCP is agent-to-tool; A2A is agent-to-agent. That distinction is exactly what a company OS needs, because internal specialists should not all be merged into one context window. They should be separate principals with separate tools, communicating through defined external contracts. A2A was originally developed by entity["company","Google","technology company"] and is now under the entity["organization","Linux Foundation","open source nonprofit"], which increases confidence that it is becoming infrastructure rather than a single-vendor feature. citeturn25view9turn27view0

For the human-facing layer, Open WebUI is one of the most useful self-hosted fronts because it is provider-agnostic, Docker-friendly, and explicitly supports Python tools, pipelines, MCP, and OpenAPI-based tool discovery. LibreChat is also a strong self-hosted option if you want a polished multi-provider chat workspace with agents, code execution, artifacts, and persistent context. For internal human team chat, Zulip is the cleanest purely open-source option among the tools surveyed; Mattermost is useful, but its own repository describes it as open core rather than purely open source. citeturn25view7turn30view0turn30view1turn30view2turn6search23turn35search1

For business records, the important finding is that mature open-source systems already model the functional reality of a company. ERPNext covers CRM, HR, payroll, recruitment, projects, orders, assets, and related operations. SuiteCRM remains useful where you want a more dedicated CRM surface with code-level customization. Nextcloud and Paperless-ngx cover file synchronization, collaboration, searchable archives, OCR, and long-term document capture. That means your AI team does not need to invent company state from scratch; it needs to operate against these systems as the source of truth. citeturn25view5turn30view10turn30view11turn34view1turn34view0

## Role model for a digital company

The requirement to consider “all possible roles” should be implemented as a **role ontology plus dynamic instantiation**, not as a finite static list of prompts. Mature business systems already break the company into stable functional domains such as CRM, HR, payroll, order management, assets, projects, records, and communications. The correct design is to map those domains into agent families, then instantiate concrete roles from manifests that define instructions, tools, memory namespaces, approval policies, and success metrics. OpenFGA’s new guidance on modeling agents as first-class principals is especially important here: agents should live inside the same authorization hierarchy as users, organizations, projects, and issues rather than floating outside the company’s permission system. citeturn25view5turn30view10turn30view11turn29view0turn29view1

A robust initial role set for a digital startup should be organized into families rather than titles. The minimum viable set is: **company builder and org architect**; **supervisor/orchestrator**; **finance and accounting**; **legal and policy**; **sales and CRM**; **marketing and public relations**; **customer support and success**; **product and project management**; **software engineering and QA**; **operations and procurement**; **people and HR**; **security and compliance**; **knowledge and research**; and **communications agents for email, chat, voice, and SMS**. In software startups, an engineering pod should also include a coding specialist such as OpenHands and a browser/app testing specialist using Playwright. These families map cleanly to the capabilities already exposed by ERP, CRM, coding-agent, browser-automation, and communications systems. citeturn25view5turn30view10turn30view11turn10search0turn30view3

The initial setup flow should be run by a **company builder** agent. That agent ingests the company profile, jurisdiction, product type, go-to-market channels, required approval rules, and internal systems; emits the first organizational blueprint; creates role manifests; and asks the human owner to approve them before activation. During live operation, unmet-capability events should trigger a **role-gap workflow**: a planner proposes a new role, MCP tool access is declared, OpenFGA relationships are defined, OPA policy checks are applied, and the new role is approved and provisioned without changing the base architecture. That is the cleanest way to make the system adaptive without letting agents arbitrarily self-escalate scope. citeturn25view9turn27view0turn29view0turn40search0turn40search7turn39view2

External communication is already well covered by open components. Asterisk is an open-source PBX and telephony toolkit, and its ARI interface exposes channels, bridges, endpoints, and media over REST plus WebSocket events so external applications can control communications directly. FreeSWITCH offers the same general class of programmable telephony through its Event Socket interfaces. For voice-agent pipelines, Pipecat and LiveKit Agents both support real-time STT-LLM-TTS orchestration and interruption handling; LiveKit also documents outbound SIP calling by creating SIP participants attached to rooms. Whisper from entity["company","OpenAI","ai company"] remains a strong open speech-recognition base, and Piper remains one of the main local TTS engines. For SMS, Jasmin is still a notable open-source gateway option when SMPP/HTTP messaging is required. The main constraint is economic, not technical: PSTN and SMS delivery still require external carriers, trunks, or providers even if the orchestration software is open source. citeturn25view6turn28view7turn18search2turn18search6turn28view5turn28view6turn13search6turn13search19turn5search19

## Memory and the illusion of infinite recall

Your instinct about memory is correct: the hard problem is not just “more context,” but **making finite-context agents behave as if they remember everything relevant**. The best pattern in the current ecosystem is a layered memory stack. Letta contributes pinned, editable in-context memory blocks and shared memory blocks that multiple agents can see immediately. LangGraph contributes thread-level checkpoints and resumable state at every step, which is essential for long workflows, human approvals, and debugging. Mem0 contributes layered conversation, session, user, and organizational memory. Graphiti adds a temporal knowledge graph that continuously updates relationships and time-sensitive facts instead of recomputing them in batches. These tools solve different memory problems and should be combined, not substituted for one another. citeturn28view0turn28view1turn28view2turn33view0turn25view2

In practice, the memory stack should have four levels. First, **pinned identity memory**: company constitution, role charter, approval matrix, and customer-specific standing instructions. Second, **workflow memory**: current thread state, tool outputs, pending approvals, and local working state. Third, **retrieval memory**: episodic notes, user preferences, durable org memory, and temporal relationships. Fourth, **canonical record systems**: invoices, contracts, tickets, payroll, CRM records, files, and audit logs. The fourth level is not “memory” in the LLM sense; it is the truth. The agent memory layers should summarize and point into canonical records, not replace them. That separation is what prevents drift, hallucinated state, and silent inconsistency. citeturn33view0turn28view2turn25view5turn29view6turn34view1turn34view0

The canonical data backbone should be PostgreSQL plus the relevant business systems. PostgreSQL gives you JSON/JSONB, LISTEN/NOTIFY for asynchronous process signaling, logical replication for controlled downstream projections, and row-level security for per-agent visibility control. Qdrant is a strong choice for semantic retrieval because it supports HNSW indexing, metadata payload indexes, filtering during search, and open-source self-hosting. Mem0’s own documentation also warns against storing secrets or unredacted PII in retrievable memory because that memory is retrievable by design; that warning is correct. Secrets belong in a dedicated secrets manager, not in a vector store or shared agent note. citeturn21search2turn34view4turn34view5turn29view6turn29view5turn29view7turn33view0turn29view2turn31view8

## Recommended open-source architecture

The strongest architecture from the current survey is the following stack.

- **Owner console and human interface:** Open WebUI as the main operator portal, because it is self-hosted, Docker-friendly, provider-agnostic, and natively extensible through Python tools, pipelines, MCP, and OpenAPI. Pair it with a custom admin view for approvals, memory inspection, and agent status; use LibreChat only if you want a more chat-centric multi-user interface rather than a control-plane-first interface. citeturn25view7turn30view0turn30view1turn30view2

- **Agent runtime:** LangGraph as the main reasoning/orchestration runtime, using subgraphs for specialist pods and interrupts for human review on sensitive actions. Keep AG2 available for swarm or group-chat experiments, and CrewAI for thin role-play prototypes, but do not make either the company’s primary execution substrate. LangGraph’s persistence and interrupt model is materially better aligned with real business operations than pure conversational swarms. citeturn25view0turn28view2turn39view1turn39view2turn31view0turn31view1turn35search7

- **Durable workflow plane:** Temporal for every business process that must survive restarts, retries, timeouts, handoffs, or manual approval waits: onboarding, sales stages, invoicing, collections, hiring workflows, vendor approvals, outbound campaigns, and incident response. Use Signals, Queries, and Updates so the owner can inspect and steer workflows from the console. Use Prefect or Activepieces only for narrower data/automation jobs, and prefer Activepieces over n8n if license cleanliness matters. citeturn25view3turn28view3turn28view4turn24search1turn30view9turn30view8turn35search2turn35search0

- **Tooling and interoperability:** Use MCP for every external tool surface—filesystems, databases, ERP, CRM, calendars, ticketing, Git, browser automation, and finance connectors. Use A2A for cross-agent delegation between specialist services. This gives you a clean split: tools remain tools; specialists remain specialists. It also makes future role creation much easier because a new role becomes a new manifest plus its MCP/A2A contracts, not a refactor of the whole company. citeturn25view9turn27view0

- **Memory plane:** Use Letta for pinned and shared editable memory; LangGraph checkpoints for workflow continuity; Mem0 for layered conversational, session, user, and org memories; Qdrant for semantic retrieval; and Graphiti for evolving temporal relationships. This combination is better than relying on only a vector database, only a prompt, or only chat history. It is the closest practical approximation to “infinite memory” without violating the reality of finite context windows. citeturn28view0turn28view1turn28view2turn33view0turn25view2turn29view7

- **System of record:** Use PostgreSQL as the transactional backbone; ERPNext as the company ERP for accounting, CRM, HR, orders, and assets; SuiteCRM selectively if sales needs exceed ERPNext’s CRM surface; Nextcloud for shared files and collaboration; Paperless-ngx for searchable document archives; docassemble for guided document assembly; and DocuSeal or OpenSign for open-source signing workflows. Record every business event in canonical systems first; then let agent memory reference and summarize those records. citeturn25view5turn30view10turn30view11turn34view1turn34view0turn38search0turn38search3turn38search9turn38search2

- **Communications plane:** Use Asterisk or FreeSWITCH for programmable voice infrastructure; Pipecat or LiveKit Agents for the realtime voice pipeline; Whisper for local or open speech recognition; Piper for local TTS; and Jasmin if you need an open-source SMS gateway. For browser and web tasks, add Playwright. For software engineering roles, add OpenHands. citeturn25view6turn28view7turn18search2turn18search6turn28view5turn28view6turn13search6turn13search19turn5search19turn30view3turn10search0

- **Model gateway, observability, and governance:** Put LiteLLM in front of all models for provider abstraction, retries, fallbacks, virtual keys, and cost tracking. This is the layer that prevents a company OS from becoming an opaque prompt box. citeturn25view8turn12search18turn12search8turn12search9turn25view4turn39view0turn24search2turn37view0turn37view1turn31view5turn31view6turn31view7turn29view3turn29view0turn40search0turn40search7turn29view2turn31view8

This stack is “better than the existing monoliths” in one specific way: it removes the most common failure modes. It avoids the **single-agent bottleneck** by separating specialists; avoids **memory drift** by separating retrieval memory from canonical records; avoids **unsafe tool execution** by putting policies, permissions, approvals, and secrets outside the prompt; avoids **silent workflow loss** by using Temporal and checkpoints; and avoids **operator blindness** by making traces, decisions, and approvals observable in human-friendly systems. citeturn27view0turn28view2turn33view0turn29view6turn29view0turn40search0turn39view2turn25view3turn25view4turn39view0

## Docker deployment and operator experience

Despite the desire for “a Docker container,” the right deployment target is a **single Docker Compose application**, not one giant process image. Docker’s own documentation explicitly treats multi-container applications as the normal pattern for larger systems and recommends avoiding “one container doing multiple things.” Compose gives you one command to launch the whole stack, dependency ordering, networks, volumes, and environment configuration, while still keeping each service independently replaceable and debuggable. For this system, that means separate containers for the UI, agent API, workflow workers, Temporal, PostgreSQL, vector store, observability, and security services. citeturn32view1turn32view4turn22search4

The terminal logging requirement is also already supported cleanly by Compose. `docker compose up` aggregates logs from attached services, and Docker documents both `--attach` and `--no-attach` so you can keep the terminal focused on a small set of high-value logs instead of flooding it with database or metrics noise. Compose profiles let you keep optional services—debugging, analytics, benchmarking, or heavy local-model servers—disabled unless needed. Compose healthchecks and dependency ordering are the right way to ensure the agent API does not start before PostgreSQL, Temporal, or the vector store are actually ready. citeturn32view0turn32view2turn32view3

The operator experience should therefore be: one Compose project, sensible healthchecks, persistent named volumes, attached logs only for the owner console plus core API/worker services, and everything else discoverable through dashboards rather than noisy stdout. That satisfies the isolation goal, keeps the terminal readable, and remains consistent with Docker’s own model for production-grade multi-service apps. citeturn32view0turn32view1turn22search14

## Risks, constraints, and open questions

The core architectural constraint is governance. This system can automate a large share of digitally-executable work, but it should not be allowed to improvise unrestricted authority. LangGraph’s interrupt and human-in-the-loop middleware already provide the right control pattern: pause execution, surface the proposed action, and allow approve, edit, reject, or respond. Pair that with OpenFGA permissions, OPA policy checks, and a secrets manager so agents only receive scoped powers, never blanket access. In practice, contracts, payments, payroll, terminations, credential issuance, irreversible data deletion, and high-risk external messaging should all cross this approval boundary. citeturn39view1turn39view2turn29view0turn40search0turn29view2turn31view8

Telecom automation needs special caution. In the United States, the entity["organization","Federal Communications Commission","us regulator"] states that autodialed texts to mobile phones generally require prior consent, and the FCC has also confirmed that TCPA restrictions apply to AI-generated human voices. If your startup operates in the U.S., outbound calling and texting must therefore sit behind policy gates, consent records, opt-out handling, disclosure rules, and complete audit trails. Equivalent constraints will exist in other jurisdictions, but the exact policy matrix must be localized before production rollout. citeturn19search0turn19search16turn19search21turn19search2

There is also a license-quality constraint. The cleanest open-source path from this research is: Activepieces core over n8n when choosing visual automation; Zulip over Mattermost if you want a clearly fully open-source internal chat platform; and open-source self-hosted infra for the core runtime and records plane. n8n is fair-code, Mattermost is open core, and some adjacent tools mix open and commercial editions. None of that disqualifies them technically, but it does mean the final bill of materials should pass a separate license review before you standardize. citeturn35search2turn35search4turn35search0turn35search1turn6search23

Open questions remain. The biggest unresolved items are the company’s jurisdiction, industry, languages, expected interaction volumes, tolerance for open-core licenses, and whether model inference should be local-first or cloud-assisted. Those choices affect legal automation depth, payroll and tax workflows, telephony compliance, hardware sizing for local models, and which roles should be created at bootstrap instead of on demand. The architecture above is the highest-confidence baseline; the exact role manifests and policy bundles still depend on those company-specific constraints.
